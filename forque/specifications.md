# Specifications: Co-mit / Forque (開発仕様書：WHYを添えて)

## 1. 核心となる思想 (WHY)
* **「創作はクリシェ（再構築）である」:**
    * **WHY:** ゼロからの完全なオリジナルを強いることは、クリエイターに過剰な心理的負荷をかけます。「既存の何かから派生する」ことを「恥」ではなく「正当な系譜」として認めることで、コラボの心理的ハードルを下げ、創作の循環を加速させます。

## 2. UI用語の抽象化 (WHY)
技術用語を隠蔽し、クリエイターが直感的に操作できる名称を採用する。

| 技術用語 | 提案名称 | なぜこの名称か (WHY) |
| :--- | :--- | :--- |
| Issue | 壁打ち・お題 | 議論を事務作業ではなく、創作に向けた「思考のラリー」と定義するため。 |
| Pull Request | 取り込み依頼 | 技術的文脈から「他者の力を借りて作品を豊かにする」という創造的文脈へ変換するため。 |
| Push | 公開・保存 | 成果を記録・発表する能動的な行為であることを強調するため。 |
| Fork | 創作の種・派生 | 既存作品が次なる創作の出発点であるというポジティブな因果関係を示すため。 |

## 3. 信頼と安全性（心理的設計） (WHY)
* **歩幅（Pace）プロフィールの導入:**
    * **WHY:** 制作速度の不一致はコラボ解消の最大の火種です。事前に「自分の歩幅」を申告させることで期待値を調整し、トラブルを未然に防ぎます。
* **ポジティブな休止（アーカイブ）:**
    * **WHY:** 「飽きた」「逃げた」という負の感情を排除するため。創作をひとつの収穫物と捉え、次のサイクルへ進む儀式化をすることで、健全なフェードアウトを促します。
* **お断りのテンプレート:**
    * **WHY:** 拒絶による罪悪感をシステムが仲介することで、「機会のミスマッチ」という事務的な事実に置き換え、精神的負担をゼロにするため。

## 4. 収益と持続可能性 (WHY)
* **多角的な収益モデル:**
    * **WHY:** 収益源を多様化することで、売上のために「質」を犠牲にすることなく、コラボの「質」を重視する文化を維持するため。
* **自動収益分配:**
    * **WHY:** 著作権や分配金の交渉は挫折の最大の要因です。GitHubの貢献度グラフを応用した自動分配により、対人ストレスを排除します。

## 5. AI共存文化 (WHY)
* **AI使用率の可視化:**
    * **WHY:** AIの利用自体を悪とせず、正直な申告を奨励する設計。乖離があるユーザーを動的に分離することで、誠実なクリエイター同士の環境を保護・育成します。

## 6. 理想のチュートリアル・サンプル
* **系譜のデモ:** 1. 絵師Aが「森のキャラ」を描く（種を蒔く）。
    2. 小説家Bが影響を受けて「短編小説」を書く（インスパイア）。
    3. ミュージシャンCが「森のテーマ曲」を投稿する（コラボ）。
    4. 批評家Dがこれらを繋ぐ「批評」を投稿し、新たなブランチが生まれる。
    * **WHY:** 初回ユーザーが「自分の作品も誰かの種になり得る」という成功体験を直感的にイメージできるようにするため。

### MVP優先機能（Minimum Viable Product）
初期検証のための最小構成。心理的安全性と創作の系譜可視化の本質を最優先に検証する。  
まずはDiscord botで低コスト・低リスクにスタートし、ユーザー検証→イテレーションを繰り返す。

#### コアMVP（必須・最初に実装・検証対象）
- Pace申告（必須項目）とアイコン表示（閲覧時の判断材料としてカメ/ウサギ/ロケットなど視覚化）
- Wall-bouncing（テーマ）公開参加（オープンなテーマ投稿欄）
- Incorporation Request（コメント欄＋必要最低限のテンプレートメッセージ）
- Declination Templates（デフォルト複数用意＋プレビュー機能）
- ポートフォリオ必須（外部URL入力＋初回作品アップロード検証）
- 年齢・性別非表示徹底（World ID推奨で裏側認証、Orbバッジオプション）
- Discord bot実装（スラッシュコマンド中心: /pace, /theme, /request, /decline, /help, /start, /export-data）
  - 初回チュートリアル自動送信
  - 導入ハードル対策（招待説明、待機時間明記）
- 金銭授受なしモード（収益分配機能オフ）

#### MVP後期〜将来拡張（検証後優先順位で実装）
- 不可視誠実度スコアによるマッチング優先
- 自動収益分配（商用申告型＋ペナルティ）
- Git-like履歴（メタデータ＋外部ストレージ参照）
- 企業コンペ形式依頼機能
- Fediverse連携（v2.0 Federatedモード）
- FORQUE規格実装（貢献グラフ共有フォーマットなど）

**MVPのゴール**：  
心理的安全性を体感できる最小の創作ループ（テーマ投下 → Request → 系譜枝分かれ → 断り/継続）をDiscord内で成立させ、ユーザー数10〜50人規模でフィードバックを集める。


## 7. Main Branch Policy（本流ポリシー）

本流（main）は現代の創作文化における「正統性」「公式性」を扱うための暫定的な概念であり、  
未来の創作文化では「本流なし（mainless）」が自然になる可能性がある。  
FORQUE はこの過渡期に対応するため、柔軟な本流ポリシーを定義する。

### 7.1 本流決定の優先順位
FORQUE における本流の扱いは、以下の優先順位で決定される。

1. **参加者全員の合意**  
2. **メインブランチ作成者の判断**  
3. **作者が指定した代理人（co-maintainer）**  
4. **継承ポリシーに従う**  
5. **本流なし（mainless）モード**

### 7.2 作者不在時の扱い（コピーレフト解放）
作者の生存判定は技術的に不可能であり、著作権法により作品が凍結する危険がある。  
FORQUE は文化的停滞を避けるため、以下の仕組みを採用する。

- 作者が設定した期間ログインがなければ、本流ポリシーの権限のみ解放される  
- 著作権は保持される  
- **期間は作者が自由に設定可能（1年〜無期限まで）**  
- FORQUE は参考値として **5年** を推奨するが、強制力はない

```json
"main-branch-policy": {
  "allow-claim-main": false,
  "conditions-for-main-claim": ["author-approval"],
  "co-maintainer": null,
  "mainless": false,
  "copyleft-release-after": "author-defined"
}

7.3 本流が複数存在するモード（multimain）
- 複数の main を同時に認める
- マルチバース型創作に対応
- コミュニティごとに異なる“本流”を持つことが可能

8. Inheritance Policy（継承ポリシー）
継承ポリシーは、作品がどのように派生され、再利用されるかを定義する。
Creative Commons のように 選択制 とし、文化的自由度を最大化する。
8.1 継承ポリシーの原則
- 派生は文化的に自然であり、原則として許可される
- クレジットは文化的義務
- 商用利用は作者の選択
- 派生ポリシーは「緩和のみ許可」（強化は禁止）
- lineage（系譜）は必ず記録する
- AI 生成部分は自己申告＋メタデータで扱う

9. Technical Vocabulary（FORQUE 技術語彙）
FORQUE は世界観語彙を含まず、技術者に伝わりやすい中立語彙のみを採用する。

10. Future-Proof Modes（未来対応モード）
FORQUE は未来の創作文化に対応するため、以下のモードを正式に規格として保持する。
10.1 mainless（本流なし）
- 本流を定義しない
- 正統性を中央集権的に扱わない
- 未来の創作文化に適応
10.2 multimain（本流複数）
- 複数の main を同時に認める
- マルチバース型創作に対応
10.3 co-maintainer（作者代理人）
- 作者不在時の文化的停滞を防ぐ
- OSS 文化と同様の継承構造を持つ

11. AI 生成部分の扱い
AI が生成した部分は、以下のメタデータで扱う。

"generated-by": "human" | "ai" | "hybrid",
"model": "optional-model-name",
"confidence": "optional-confidence-score"


- AI が main を作る未来にも対応
- 自己申告を基本とし、強制的な判定は行わない
- lineage に AI の関与を記録することで透明性を確保

12. Summary
本追記は、Copilot セッションで確立した以下の要素を specifications.md に統合する：
- 本流ポリシーの体系化
- 継承ポリシーの体系化
- 作者不在時の扱い（自由度の高いコピーレフト解放）
- 本流なし / 本流複数 / 代理人
- AI 生成部分の扱い
- 技術語彙の明確化
- 未来対応モードの規格化
これにより、FORQUE は
現代の創作文化と未来の創作文化の両方に対応する規格として成立する。
