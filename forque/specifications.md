## 1. 核心となる思想

**「創作はクリシェ（再構築）である」**

ゼロからの完全なオリジナルを強いることは、クリエイターに過剰な
心理的負荷をかける。「既存の何かから派生する」ことを「恥」ではなく
「正当な系譜」として認めることで、コラボの心理的ハードルを下げ、
創作の循環を加速させる。

---

## 2. UI用語（三層構造）

forqueは世界観・直感・技術の三層で用語を使い分ける。

| 技術用語 | UI（一般） | ドキュメント | 規格 |
|---|---|---|---|
| Fork（動詞） | 派生する | 枝分かれ | fork |
| Pull Request | つながり申請 | 取り込み依頼 | pull-request |
| Push | 公開する | 公開 | push |
| Branch | 枝 | 枝 | branch |
| Main Branch | 幹 | 幹／本流 | main-branch |
| Issue | 投げかけ | 投げかけ | issue |
| Lineage | 家系図を見る | 系譜 | lineage |
| Merge | 合流 | 合流 | merge |

WHY: 技術用語を隠蔽し、クリエイターが直感的に操作できる名称を採用する。
世界観語彙は文化を育て、技術語彙は規格を普遍化する。

## 3. Pace（歩幅）設計

### 3.1 Paceが表すもの
複合的なテンポ感の自己申告。以下の3軸を総合して選ぶ：
- 返信の速さ（DM・申請への返答）
- 制作のペース（作品を仕上げる頻度）
- 関与の密度（プロジェクトにどれくらい深く入れるか）

WHY: 単一軸では実態と乖離するケースが多い。
「だいたいこのくらい」という自己認識を申告する設計。

### 3.2 段階
| アイコン | ラベル | 英語名 |
|---|---|---|
| 🐢 | ゆっくり | Slow |
| 🐇 | ふつう | Steady |
| 🚀 | 全力 | Sprint |

WHY: 3段階を超えると判断が複雑になる。速さの優劣ではなく、
等しく正当なテンポとして扱う。

### 3.3 粒度（二層構造）
- ユーザーデフォルト: アカウントページに表示
- プロジェクト上書き可: プロジェクト作成時に1タップで設定

プロジェクトページの表示ルール:
- ユーザーPaceとプロジェクトPaceが同じ場合: プロジェクトPaceのみ表示
- 異なる場合: 両方表示＋説明文
  （例:「このプロジェクトでは普段より速く動いています」）

WHY: 人によって作品ごとにペースが異なる現実に対応しつつ、
申告の手間を最小化する。

### 3.4 変更インターフェイス
- プロジェクト作成時: 1タップで選択するフローを挟む
  （デフォルトはユーザーPaceを表示）
- 変更時: アイコンをタップ → 即切り替え（確認ダイアログなし）
  → トースト通知のみ
- WHY: 変更の心理的ハードルを最小化する

### 3.5 休眠判定
- 90日未ログインでPaceアイコンが薄くなる（1段階のみ・二値判定）
- 薄くなったアイコンに 🫘（休眠種）オーバーレイを重ねる
- 90日は仮採用、実装時要調整
- WHY: forqueは「遅い人も歓迎」のサービス。厳しすぎる判定は哲学に反する

## 4. 申告一致率（旧: 誠実度スコア）

- 存在は開示・仕組みは非開示（方針B）
- 説明文:「forqueは申告内容と実際の行動の一致を、
  コミュニティの健全性の指標として静かに観察しています」
- レコメンド連動: つながり申請時の参考表示 ✅ / トップページ優先表示 ❌

WHY: プロセスの種類によってAI関与の性質が異なるため数値は無意味。
完璧なシステムを目指さない。誠実に謝れる文化を支援する。

---

## 5. AI使用率申告

数値（パーセンテージ）申告を廃止し、プロセス層申告を採用。

- 使用した工程: 企画 / 下書き / 仕上げ / 翻訳 / その他
- 使用ツール（任意）
- 人間の主体性: 主導 / 協働 / 補助的

メタデータ例:
```json
{
  "generated-by": "hybrid",
  "process": ["ideation", "finishing"],
  "human-agency": "lead"
}
```

WHY: プロセスの種類によってAI関与の性質が異なるため、
数値での一括申告は実態を反映しない。

## 6. ビジュアル設計

植物＋星座のハイブリッドメタファー:

| ブランチ状態 | 植物 | 星座 |
|---|---|---|
| 新規・派生元なし | 🌱 種 | ● 単独の星 |
| 派生が発生している | 🌿 芽・茎 | ●—● 線が引かれた星 |
| 複数から合流 | 🌸 花 | ✦ 輝点 |
| 孤立（未接続） | 🫘 休眠種 | ● 点滅する星 |
| mainless全体 | 🌲 森 | 星座全体 |

設計原則:
- 「枯れた」「死んだ」アイコンは使わない
- 孤立ブランチは「接続待ちの可能性」として表示
- 非公開（下書き）は家系図に表示しない

## 7. 派生フロー（4パターン＋3.5）
```
【パターン1】公開前 → 申請不可（公開を促すガイド表示）

【パターン2】公開後 → 申請 → 承認 → 家系図に接続

【パターン3】公開後 → 申請 → 却下 → 孤立ブランチとして存在
             └ 引き取りオプション:
               A. 孤立継続
               B. 独立プロジェクトとして再出発
               C. 非公開

【パターン3.5】公開後 → 申請 → 90日無返答
               → ポリシーに応じた選択肢を申請者に提示:
               ・孤立ブランチ許可ポリシー:
                 「しばらく連絡が取れていないようです。
                  このまま待つか、孤立ブランチとして
                  独立させることもできます。」
               ・系譜から切断ポリシー:
                 「しばらく連絡が取れていないようです。
                  このまま待つこともできます。」
               ・削除要求可ポリシー:
                 上に加え「申請を取り下げることもできます。」

【パターン4】ポリシー許可 → 「派生する」ボタン押下
             → 自動チェック → 即時接続
```

WHY: Paceと同じ「期待値の事前調整」の原則を派生フローにも適用。
孤立ブランチを「失敗」ではなく「可能性」として扱う。

## 8. 継承ポリシー

メインブランチ作成時に設定:

| 設定項目 | 選択肢 |
|---|---|
| 派生の可否 | 許可（申請不要）/ 許可（承認制）/ 不可 |
| 却下後の扱い | 孤立ブランチとして存在を許可 / 系譜から切断 / 作品ごと削除要求可 |
| ポリシーの可視性 | 「派生する」ボタン押下前に表示 |

原則:
- 派生は文化的に自然であり、原則として許可される
- クレジットは文化的義務
- 商用利用は作者の選択
- 派生ポリシーは「緩和のみ許可」（強化は禁止）
- lineage（系譜）は必ず記録する
- AI生成部分は自己申告＋メタデータで扱う

---

## 9. 合流（merge）ポリシー

- 異なるプロジェクト同士の合流も可能
- 継承ポリシーが矛盾する場合は厳しい方が優先（自動統一）
- ポリシー競合時はシステム警告＋作者の明示的承認が必要

---

## 10. Main Branch Policy（本流ポリシー）

### 10.1 本流決定の優先順位
1. 参加者全員の合意
2. メインブランチ作成者の判断
3. 作者が指定した代理人（co-maintainer）
4. 継承ポリシーに従う
5. 本流なし（mainless）モード

### 10.2 作者不在時の扱い
- 作者が設定した期間ログインがなければ、本流ポリシーの権限のみ解放
- 著作権は保持される
- 期間は作者が自由に設定可能（1年〜無期限まで）
- FORQUE参考値: 5年（強制力なし）
```json
"main-branch-policy": {
  "allow-claim-main": false,
  "conditions-for-main-claim": ["author-approval"],
  "co-maintainer": null,
  "mainless": false,
  "copyleft-release-after": "author-defined"
}
```

### 10.3 未来対応モード
- **mainless**: 本流を定義しない。正統性を中央集権的に扱わない
- **multimain**: 複数のmainを同時に認める。マルチバース型創作に対応
- **co-maintainer**: 作者不在時の文化的停滞を防ぐ代理人制度

## 11. お断りテンプレート

設計原則:
- 主語は「私の状況」（相手の作品を評価しない）
- 「今回は」を必ず入れる
- 感謝は一文まで

| | 日本語 | 英語 |
|---|---|---|
| A（タイミング） | 「いまの自分の歩幅では、このプロジェクトに誠実に向き合える状況ではありません。機会がずれてしまったようです。あなたの作品が誰かに届くことを願っています。」 | "My current pace doesn't allow me to engage with this project the way it deserves. The timing just doesn't align right now. I hope your work finds the right collaborator." |
| B（方向性） | 「拝見しました。今回は私の創作の方向と重なりが少ないと感じました。お声がけいただいたこと自体、ありがたく受け取っています。」 | "Thank you for reaching out. After looking through your work, I feel our creative directions don't overlap enough this time. I genuinely appreciate the invitation." |
| C（キャパシティ） | 「現在進行中のプロジェクトがあり、新しいコラボへのエネルギーをうまく用意できません。状況が変わったときにまた考えさせてください。」 | "I have ongoing projects that are using my full bandwidth right now. I can't take this on with the care it deserves. Let's keep the door open for another time." |
| D（孤立ブランチ通知・システム送信） | 「今回は系譜をつなげることができませんでした。あなたの作品はここに独立した枝として残ります。別の誰かが手を伸ばすかもしれません。」 | "We couldn't connect the lineage this time. Your work remains here as its own branch — someone else may reach for it someday." |

## 12. homage／盗作判定ガバナンス

### MVP期
- flagのみ実装
  - flagにはコスト: 回数制限・理由必須・虚偽flagペナルティ
- 「議論ありバッジ」表示・判定なし
- タイムスタンプ自動防御（flagした側が新しければ自動却下）

### v1.0
```
陪審員: 3〜5名
  - 系譜2〜3段階離れ・当該ブランチ未参加・ブランチごとに1票
AI参考人（複数・投票権なし）:
  - 類似度分析AI: 線・構図・テキストの技術的一致度
  - メタデータAI: タイムスタンプ・制作履歴の整合性
  - 判例AI: 過去の類似ケースを提示
人間参考人: 当事者双方が各1名指名可（投票権なし・テキスト一回限り）
```

陪審の熟慮促進設計:
- コンテンツ種別で期限自動調整（イラスト7日/短編14日/長編30日）
- 最低閲覧時間を計測・短すぎる場合は確認を促す（強制なし）
- 判断理由の記述必須（一文以上）
- 陪審員の判断履歴を記録（覆った判断が多い場合は次回選出されにくい）
- 3回連続放棄 → 申告一致率にペナルティ（長期未ログインは除外）

## 15. MVP戦略

### 方針
- **Web先行MVP**（Discord botから方針転換）
- WHY: ターゲット層（イラスト・小説・音楽）はDiscord
  コマンド操作に不慣れ。DiscordはiFrame不可でイラスト・
  音楽の「見せる」体験が困難

### ランディングページ（先行実施）
- サービス本体の前に1ページだけ公開して需要を検証
- 掲載内容: 核心的価値観3点＋メールアドレス登録＋登録人数表示
- 実装手段: Carrdなど技術不要のツール

### Discord bot（並走・概念検証用）
- `/start`の1コマンドのみ覚えれば、以降はボタン操作で完結
- ゲーマー・技術系ユーザーへの概念検証として並走させる位置づけに
- WHY: 彼らの反応がターゲット層を代表しない点に注意

**`/export-data` コマンド**
- エクスポート対象: lineage・ポリシー設定・AI使用申告メタデータ等
  FORQUE規格データのみ
- 作品本体は外部サービス（pixiv・SoundCloud等）に存在するため対象外
- 出力形式: FORQUE規格JSON
- 用途: Fediverse移行・サービス間データ移動・自己データのバックアップ
- WHY: forqueはコンテンツをサーバーに保存しない方針。
  ユーザーが「自分のforque上の履歴」を持ち出せることを保証する

### MVPコア機能
- Pace申告と表示
- フォーク申請・お断りテンプレート
- ポートフォリオ必須（外部URL入力）
- 年齢・性別非表示徹底
- 金銭授受なしモード

## 16. オンボーディング設計

**基本方針: デモシナリオ（C）＋使いながら覚える（B）の組み合わせ**

フロー:
1. デモシナリオ（「森の声」: Amara/Berk/Celia/Dani）を30秒で見せる
2. Pace申告だけさせる（最初のアクションは1つ）
3. 作品を1つ登録させる（外部URLまたはアップロード）
4. 以降は自然に流れる（フォーク申請が来たら通知＋そのとき説明）

WHY: 一度に全部教えない。最初の体験を1アクションに絞る。

### 「？」ヘルプインターフェイス
- 汎用ヘルプ機能として設計
- 用途: Paceの3軸ガイド・用語説明（glossary連携）
- 必要なときだけ参照できる。説明を押しつけない
