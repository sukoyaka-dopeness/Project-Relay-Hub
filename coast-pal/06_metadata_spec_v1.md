CoastPals：メタデータ構造定義書 (v1.0)

本ドキュメントは、ボトルメールが海を漂う際に付与されるデータの全容を定義する。これまでのリレーで議論された「匂い」「劣化」「動物イベント」をすべて論理データとして構造化し、MVPと将来フェーズの境界を明確にしたものである。

1. メタデータ設計の原則

秘匿性: ユーザーUIには「具体的な数値」は出さず、すべて文学的・情緒的な表現に変換して表示する。

永続性: 将来のNFT化（レジェンダリー認定）に備え、すべての旅の履歴を配列として保持する。

動的変化: 海流、時間、動物との接触により、データは常に上書きではなく「追記」される。

2. コア・データ構造 (JSON Schema)

2.1 共通ヘッダー (Origin Data)

ボトルの投函時に一度だけ生成されるデータ。

bottle_id: UUID (一意の識別子)

sender_id: 送信者のハッシュID

timestamp_origin: 投函時刻 (ISO 8601)

location_origin: 投函海域 (例: mediterranean, polynesian)

initial_smell: 投函時の匂いカテゴリ

2.2 漂流メタデータ (Journey Metadata)

ボトルの状態を示す動的な値。

current_degradation: 0.0〜1.0 (MVPでは0.8でキャップ)

total_distance_km: 累積移動距離

elapsed_days: 投函からの経過日数

current_smell: 現在の匂い (動物接触や時間で変化)

re-arrival_weight: 再漂着（縁）の補正値 (初期+5%、返信成功で最大8%)

2.3 履歴配列 (Journey Logs)

NFT化の際の「物語」の源泉となる配列。

history_logs: 以下のオブジェクトの配列

event_type: landfall (漂着), animal_touch (動物接触), current_shift (海流変化)

event_detail: 例: cat_played, goat_nibbled, penguin_speedup

timestamp: イベント発生時刻

location_tag: 発生した海域

3. 実装フェーズ別機能区分

開発およびハンドオフの混乱を防ぐため、機能をフェーズ別に分離する。

3.1 MVP実装（即時実装対象）

テキスト劣化: シンプルな文字欠損（文字化け、スペース置換、アスタリスク置換など）。

匂いシステム: 投函時に選択またはランダム付与される3〜5種の固定タグ。

動物イベント: 猫（遅延）、ペンギン（速達）、ヤギ（劣化加速）の3種。

地理概念: 仮想的な4大海域（地中海、ポリネシア、南米、アフリカ）のタグ付与。

時刻ロック: 送信から受信可能になるまでの物理的な待機時間（3〜6時間）。

3.2 将来拡張（Post-MVP / 後期フェーズ）

高度な劣化と翻訳: 言語特性（音節など）に応じた劣化、およびGemini APIを活用した「あえて不完全な」自動翻訳。

AI匂い推定: AIがメッセージ内容を解析し、最適な匂いタグを自動付与。

追加生物: 深海魚（サルベージ）、クジラ（長距離輸送）、渡り鳥（国境越え）などのイベント。

リアル海流連携: 実際の海洋気象データに基づいた移動シミュレーション。

称号システム: 旅の履歴（履歴配列）を解析して生成される「地球3周」などのメタデータ称号。

NFT化機能: 完結した旅の記録を永続化するためのオンチェーン連携。

4. 時刻と「待ち」のロジック

最小漂流時間: 送信から受信可能になるまで、システム的に「最低3〜6時間」のロックをかける。

補正: 送信者と受信者の間に「縁（再漂着補正）」がある場合、このロック解除後に優先的にマッチングリストへ浮上させる。

理由: 「デジタルなのに待たされる」という情緒を物理的なフラグ（is_drift_complete）で管理するため。

5. Gyres（海流の渦）の仕様

「海流の環（Gyres）」は、受信したボトルを整理するユーザー側のフィルタ・コレクション機能である。

ユーザーは独自の「渦（フォルダに代わる概念）」を作成できる。

特定の「匂い」や「海域」を持つボトルを、その渦に自動または手動で吸い込ませる。

仕様の本質: 整理整頓ではなく、「同じ傾向を持つ言葉たちが、自分の手元で再び混ざり合う」という体験を提供する。

6. 技術的負債の防止（AIリレーへの戒め）

次段階のAIおよび開発者は、以下の項目を 「変更不可（Locked）」 として扱うこと。

1送信につき1受信: 1対多の配信は行わない。

DM機能の排除: ボトルメール以外の連絡手段を設けない。

生数値のUI露出禁止: ユーザーには常に「感触」として伝える。
